01 Project Overview

What CodeCards is, why it was built, and what a user can do with it.

What is CodeCards?

CodeCards is a full-stack web application that lets learners create digital flashcard decks focused on programming syntax, concepts, and code snippets. Users can build personalised decks, add cards with a "question" front and "answer" back, then quiz themselves and track their progress over time.

🃏

Flashcard Decks

Organise cards into language-tagged decks (JavaScript, Python, SQL…). Add a title and description.

💻

Code Cards

Cards can hold plain text or monospaced code snippets, with a language hint for context.

🧠

Quiz Mode

Cards are shuffled, presented face-down. Flip to reveal the answer, then mark Got it / Missed it.

📊

Score History

Every completed quiz saves a score. The dashboard shows the last 20 results with % ratings.

🔐

User Accounts

Register and log in with email + password. All decks and scores belong privately to each user.

📱

Responsive Design

Works on desktop, tablet and mobile. Clean dark-themed UI purpose-built for code content.

Who is it for?

Students studying programming, self-taught developers learning new languages, bootcamp attendees who want a better way to memorise syntax, and anyone who wants to replace paper/Anki-style flashcards with a purpose-built coding tool.

04 Architecture & Data Flow

How the pieces connect end-to-end.

High-Level Architecture

┌─────────────────────────────────────────────────┐ │ BROWSER (Frontend) │ │ index.html / pages/*.html │ │ js/api.js → fetch() with JWT header │ └──────────────────────┬──────────────────────────┘ │ HTTPS (JSON) ▼ ┌─────────────────────────────────────────────────┐ │ BACKEND (Node.js / Express) │ │ │ │ server.js → index.js (app) │ │ ├── helmet() (security headers) │ │ ├── cors() (origin whitelist) │ │ ├── express.json() (parse request body) │ │ │ │ │ ├── /api/auth → auth.routes → auth.ctrl │ │ ├── /api/decks → deck.routes → deck.ctrl │ │ ├── /api/decks/:id/cards → card.routes │ │ └── /api/quiz → quiz.routes → quiz.ctrl │ │ │ │ authMiddleware verifies JWT on protected routes │ └──────────────────────┬──────────────────────────┘ │ Mongoose ODM ▼ ┌─────────────────────────────────────────────────┐ │ MongoDB Atlas (cloud) │ │ collections: users decks cards scores │ └─────────────────────────────────────────────────┘

Request Lifecycle (authenticated endpoint)

1
Browser calls api.js fetch wrapper

The wrapper reads the JWT from localStorage and attaches it as an Authorization: Bearer <token> header.

2
Express receives the request

helmet, cors, and express.json() middleware run first. The matching router is selected.

3
authMiddleware verifies the JWT

jsonwebtoken.verify() checks signature and expiry. If valid, req.userId is set. If invalid, a 401 is returned immediately.

4
Controller executes the business logic

Uses req.userId to scope all database queries to the authenticated user. Calls Mongoose methods.

5
Mongoose talks to Atlas

The query is serialised over a pooled TLS connection. Atlas executes it and returns the result.

6
Controller sends JSON response

On success: res.json(data). On error: next(err) bubbles to the global errorHandler which sends { error: "..." }.

7
Frontend renders the result

The api.js wrapper resolves the promise. The page's JS function inserts HTML into the DOM.

07 Authentication System

How passwords are hashed, how tokens are issued, and how protected routes verify identity on every request.

Registration Flow

auth.controller.js — register()
async function register(req, res, next) {
  const { username, email, password } = req.body;

  // 1. Validate that all fields are present
  if (!username || !email || !password) { return next(badRequest()); }

  // 2. Prevent duplicates
  const existing = await User.findOne({ $or: [{ email }, { username }] });
  if (existing) { return next(conflict()); }

  // 3. Hash the password — cost factor 12 (~250ms per hash)
  const passwordHash = await bcrypt.hash(password, 12);

  // 4. Persist the user
  const user = await User.create({ username, email, passwordHash });

  // 5. Issue a JWT — signed with the server's secret
  const token = jwt.sign({ userId: user._id }, process.env.JWT_SECRET,
                           { expiresIn: process.env.JWT_EXPIRES_IN });

  res.status(201).json({ token, user: { id: user._id, username, email } });
}

JWT Middleware

auth.middleware.js
function authMiddleware(req, res, next) {
  const authHeader = req.headers.authorization;

  // Must be "Bearer <token>"
  if (!authHeader || !authHeader.startsWith('Bearer ')) {
    return res.status(401).json({ error: 'No token provided' });
  }

  const token = authHeader.split(' ')[1];

  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    req.userId = decoded.userId;  // available to every controller after this
    next();
  } catch {
    res.status(401).json({ error: 'Invalid or expired token' });
  }
}

Frontend Token Management

api.js — helpers
// Persist after login/register
function saveAuth(token, user) {
  localStorage.setItem('cc_token', token);
  localStorage.setItem('cc_user', JSON.stringify(user));
}

// Attach token to every API request automatically
async function apiFetch(path, options = {}) {
  const token = localStorage.getItem('cc_token');
  const headers = { 'Content-Type': 'application/json' };
  if (token) headers['Authorization'] = `Bearer ${token}`;

  const res = await fetch(`${API_BASE}${path}`, { ...options, headers });
  const data = await res.json();

  if (!res.ok) throw new Error(data.error || `Request failed`);
  return data;
}
Token storage trade-off: localStorage is used here for simplicity. In a higher-security context, httpOnly cookies would be preferred since they can't be read by JavaScript at all, protecting against XSS.
08 Frontend Pages

Every HTML page, what it does, and how the user navigates between them.

Navigation Map

Page Descriptions

Landing page — index.html

Marketing/splash page. Shows the tagline, feature description, and CTA buttons. If a JWT is already in localStorage the user is immediately redirected to the dashboard (handled by main.js).

Register — pages/register.html

Collects username, email, password. Validates password length (≥8 chars) client-side before calling POST /api/auth/register. On success, saves the token and redirects to the dashboard.

Login — pages/login.html

Collects email and password. Calls POST /api/auth/login. On success, saves the token and user object, redirects to dashboard.

Dashboard — pages/dashboard.html

When a user first logs in, this is the page they land on. At the top they are greeted by name, and their entire deck collection is laid out in a responsive card grid below. Each deck card displays the language tag, the deck title, and a short description, and carries three action buttons — Open (goes to the deck detail page), Edit (opens a modal form to rename or update the deck), and Delete (asks for confirmation, then permanently removes the deck). A floating "+ New Deck" button in the top-right corner opens the same creation modal. Below the deck grid, the page shows a table of the user's last 20 quiz scores, including the deck name, how many cards they got right, the percentage, and the date taken.

Deck Detail — pages/deck.html?id=<deckId>

This page is reached by clicking Open on any deck card in the dashboard. The deck's ID is passed in the URL as a query parameter (e.g. ?id=66abc...), which the page reads on load to fetch the correct data from the API. The header area shows the deck's language badge, title, and description, alongside a ▶ Start Quiz button that links directly to the quiz page for this deck. The body of the page lists every card in the deck, numbered, each showing both the front (the question or prompt) and the back (the answer). Cards marked as code type render their content in a monospaced dark box to make code easy to read. An "+ Add Card" button opens a modal where the user can write a new card's front, back, choose text or code type, and optionally specify a language hint. Existing cards can be edited or deleted via the action buttons on each card row.

Quiz — pages/quiz.html?id=<deckId>

The quiz page fetches the deck's cards from the server via GET /api/quiz/:deckId, which returns them in a randomly shuffled order so every quiz session feels fresh. Cards are shown one at a time as a large flashcard — only the front (the question) is visible at first. A progress bar at the top shows how far through the deck the user is. When the user is ready to see the answer, they click the card or press Space and the card performs a smooth 3D flip animation to reveal the back. Two verdict buttons then appear: Got it (keyboard: → or 1) and Missed it (keyboard: ← or 2). The user's choice advances to the next card. After the final card, a results screen shows the total score, a colour-coded percentage bar, and a short motivational message. The result is automatically saved to the database via POST /api/quiz/:deckId/score so it appears in the dashboard's score history.

09 Key Code Walkthroughs

Line-by-line explanations of the most important pieces of the codebase.

Owner-Scoped DB Queries

Every database query in the deck and card controllers filters by owner: req.userId. This is the primary security control that prevents one user from reading or modifying another user's data. 

deck.controller.js — getDeck()
async function getDeck(req, res, next) {
  // req.userId is set by authMiddleware after verifying the JWT
  const deck = await Deck.findOne({
    _id:   req.params.id,
    owner: req.userId   // ← CRUCIAL — the deck must belong to this user
  });

  if (!deck) {
    // Returns 404 whether the deck doesn't exist OR belongs to someone else
    // This avoids leaking information about other users' data
    const err = new Error('Deck not found');
    err.statusCode = 404;
    return next(err);
  }
  res.json(deck);
}

Fisher-Yates Card Shuffle

The quiz endpoint shuffles the deck's cards using the Fisher-Yates algorithm — statistically uniform and far better than sort(() => Math.random() - 0.5). 

quiz.controller.js — getQuizCards()
const cards = await Card.find({ deck: deck._id });

// Fisher-Yates shuffle: O(n), each permutation equally likely
for (let i = cards.length - 1; i > 0; i--) {
  const j = Math.floor(Math.random() * (i + 1));
  [cards[i], cards[j]] = [cards[j], cards[i]];  // ES6 destructuring swap
}

res.json({ deck, cards });

Global Error Handler

Controllers never res.status(500).json() directly. Instead they call next(err). Express routes the error to this single handler. 

middlewares/errorHandler.js
function errorHandler(err, req, res, next) {
  const statusCode = err.statusCode || 500;
  const message    = err.message    || 'Internal server error';

  // Only log full stack traces in development
  if (process.env.NODE_ENV !== 'production') {
    console.error(err.stack);
  }

  res.status(statusCode).json({ error: message });
}

Defensive HTML Escaping

Any user-generated content inserted into the DOM uses the escHtml() helper to prevent Cross-Site Scripting (XSS) attacks. 

dashboard.js + deck.js — escHtml()
function escHtml(str) {
  return String(str)
    .replace(/&/g,  '&amp;')
    .replace(/'&lt;')
    .replace(/>/g,  '&gt;')
    .replace(/"/g,  '&quot;');
}

// Usage — safe innerHTML insertion:
grid.innerHTML = decks.map(d => `<h3>${escHtml(d.title)}</h3>`).join('');
10 Quiz Engine

The entire quiz runs in the browser. No server round-trips happen between cards — only at start (fetch cards) and end (submit score).

State Machine

LOADING ──▶ CARD FRONT ──(flip)──▶ CARD BACK │ ┌─ got it ──────┤ └─ missed it ───┘ │ more cards? ──▶ CARD FRONT no more? ──▶ RESULTS

CSS 3D Flip Animation

The card flip is achieved purely with CSS transforms — no JavaScript animation library needed.

/* The outer container sets the 3D perspective */

  perspective: 1000px;
}

/* The inner wrapper preserves 3D context for children */
.flashcard-inner {
  transform-style: preserve-3d;
  transition: transform 0.45s cubic-bezier(0.4, 0, 0.2, 1);
}

/* Adding .flipped rotates the inner wrapper 180° */
.flashcard.flipped .flashcard-inner {
  transform: rotateY(180deg);
}

/* The back face is pre-rotated so it reads correctly when flipped */
.flashcard-back {
  backface-visibility: hidden;
  transform: rotateY(180deg);   /* hidden until parent flips */
}

11 Security Considerations

An overview of what protections are in place and what to be aware of.

What is protected

ThreatMitigation
Password storagebcrypt with cost factor 12 — even if the DB leaks, hashes can't be reversed quickly
Session theft via XSSAll user content is escaped with escHtml() before DOM insertion
Broken access controlEvery DB query filters by owner: req.userId — users can't access each other's data
Clickjackinghelmet() sets X-Frame-Options: SAMEORIGIN
MIME sniffinghelmet() sets X-Content-Type-Options: nosniff
Forged tokensJWTs are signed with a secret only the server knows; tampering is detected
Expired sessionsJWTs expire after 7 days; jwt.verify() rejects expired tokens
Brute-force loginexpress-rate-limit package is installed and ready to be applied to /api/auth
Secret leakageSecrets live in .env which is excluded from git via .gitignore

Known limitations (future improvements)

The most notable limitation is that the JWT is stored in localStorage, which means any JavaScript running on the same page can technically read it. In a higher-security application, the token would be stored in an httpOnly cookie instead — a special kind of cookie that the browser will never expose to JavaScript, making it invisible to any malicious scripts that might somehow end up on the page.

There is currently no email verification step on registration, which means anyone can create an account with a fake or someone else's email address. Adding a verification email before activating the account would close this gap. Similarly, the express-rate-limit package is already installed in the project but has not yet been applied to the login endpoint — wiring it in would prevent automated scripts from trying thousands of passwords in quick succession (a brute-force attack). That single change is highlighted in the callout below because it is the quickest security improvement available.

Cross-Site Request Forgery (CSRF) protection is not needed in this architecture because the app uses JWT tokens sent in the Authorization header rather than cookies — CSRF attacks specifically exploit automatic cookie sending, so they do not apply here.

Before going to production: wire express-rate-limit onto POST /api/auth/login to prevent brute-force password attacks. The package is already installed; it just needs to be applied.
12 Extending the Project

Practical ideas for how to build on top of what's here.

Suggested next features

Add rate limiting to auth (quick win)

index.js — add rate limiting
const rateLimit = require('express-rate-limit');  // already installed

const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,  // 15 minutes
  max: 20,                   // max 20 requests per window
  message: { error: 'Too many requests, please try again later' },
});

app.use('/api/auth', authLimiter, require('./routes/auth.routes'));

Spaced-repetition scoring

Instead of a simple correct/incorrect count, track per-card performance and use the SM-2 algorithm (the basis of Anki) to schedule the next review date for each card. Add an interval and nextReview field to the Card model.

Deck sharing / public decks

Add a isPublic: Boolean field to the Deck model. Create a GET /api/decks/public endpoint that returns all public decks regardless of owner. Add a "Duplicate to my account" feature.

Search

Add a MongoDB text index on card.front and card.back:

cardSchema.index({ front: 'text', back: 'text' });

Load highlight.js from a CDN and call hljs.highlightAll() after rendering code cards. The language field on each card is already designed to pass as the language hint.
